# AI 讨论助手 - UI 重构产品需求文档

**版本**: v1.0
**日期**: 2026-03-08
**状态**: 待评审

---

## 1. 背景与问题

### 1.1 当前问题

当前实现采用"点击进入讨论详情页"的设计，存在以下问题：

1. **跳转打断流程**: 用户需要离开主界面进入专门的讨论页面，造成上下文切换
2. **状态不同步**: 讨论页面显示"讨论中"，但后台可能已停止运行
3. **认知负担**: 用户难以理解"正在进行"与"已完成"的状态差异
4. **操作效率低**: 查看进度、查看历史、启动新讨论需要在不同页面间切换

### 1.2 核心洞察

**AI 多模型讨论 = 后台批处理任务**

类比参考：
- Vercel Deployment（部署进度监控）
- GitHub Actions（工作流运行状态）
- Figma 导出（后台任务 + 通知）

用户真正需要的是：
1. **任务启动入口** - 快速开始新讨论
2. **进度监控面板** - 实时了解运行状态
3. **结果操作中心** - 便捷获取输出文档

---

## 2. 产品目标

### 2.1 核心目标

将"页面跳转式"体验重构为"仪表盘监控式"体验，让用户在主界面即可完成所有操作。

### 2.2 成功指标

| 指标 | 当前 | 目标 | 测量方式 |
|------|------|------|----------|
| 启动讨论到查看结果的操作步骤 | 4步 | 2步 | 用户流程分析 |
| 界面切换次数 | 3次 | 0次 | 埋点统计 |
| 用户困惑度（状态理解） | 高 | 低 | 用户反馈 |

## 3. 用户流程设计

### 3.1 主流程

```
┌─────────────────────────────────────────────────────────┐
│  1. 输入需求 → 2. 开始讨论 → 3. 实时监控 → 4. 获取结果   │
└─────────────────────────────────────────────────────────┘
         ↓              ↓              ↓             ↓
   [主界面输入]   [底部面板展开]  [实时进度更新]  [一键导出]
```

### 3.2 详细流程

#### 流程 A: 启动新讨论
1. 用户在主界面输入需求
2. 选择模型和讨论模式
3. 点击"开始讨论"
4. **底部控制面板自动展开**，显示实时进度
5. 用户可继续留在主界面进行其他操作

#### 流程 B: 监控进行中讨论
1. 主界面显示"活跃讨论"列表卡片
2. 底部面板显示当前运行讨论的实时输出
3. 点击任意活跃讨论，面板切换到对应讨论详情
4. **无需跳转页面**

#### 流程 C: 获取已完成结果
1. 活跃讨论卡片变为"已完成"状态
2. 面板显示"查看结果"、"生成文档"、"重新讨论"按钮
3. 一键点击即可获取 PRD 文档

### 5.2 视觉设计

#### 5.2.1 色彩规范

**主色调**:
- 背景: `#f9fafb` (Gray-50)
- 卡片背景: `#ffffff`
- 主强调色: `#0ea5e9` (Sky-500) - 进行中状态
- 成功色: `#10b981` (Emerald-500)
- 警告色: `#f59e0b` (Amber-500)
- 错误色: `#ef4444` (Red-500)

**状态指示**:
- 进行中: 蓝色脉冲动画点 `●`
- 已完成: 绿色勾选 `✓`
- 等待中: 灰色圆点 `○`
- 错误: 红色感叹号 `!`

#### 5.2.2 动画规范

**进度条**:
- 过渡: `transition-all duration-500 ease-out`
- 条纹动画: 持续流动的斜条纹效果

**状态切换**:
- 使用 Framer Motion `layout` 属性
- 弹簧物理: `type: "spring", stiffness: 100, damping: 20`

**打字机效果**:
- 字符间隔: 30ms
- 光标闪烁: 1s infinite

### 5.3 响应式适配

**桌面端 (>1024px)**:
- 主内容区: 最大宽度 960px，居中
- 底部面板: 固定高度 280px

**平板端 (768px - 1024px)**:
- 底部面板: 固定高度 240px
- 模型卡片: 2列网格

**移动端 (<768px)**:
- 底部面板: 全屏抽屉式 (从底部滑出)
- 模型卡片: 垂直堆叠

## 6. 技术实现要点

### 6.1 状态管理

```typescript
// 核心状态结构
interface AppState {
  // 讨论列表
  discussions: Discussion[];

  // 当前选中讨论的ID
  activeDiscussionId: string | null;

  // 底部面板状态
  panelState: 'collapsed' | 'expanded' | 'fullscreen';

  // 全局设置
  settings: AppSettings;
}

interface Discussion {
  id: string;
  title: string;
  status: 'running' | 'paused' | 'completed' | 'error';
  progress: number;
  currentRound: number;
  totalRounds: number;
  models: ModelStatus[];
  messages: Message[];
  createdAt: Date;
  updatedAt: Date;
}

interface ModelStatus {
  modelId: string;
  name: string;
  status: 'pending' | 'running' | 'completed' | 'error';
  progress: number;
  response?: string;
  error?: string;
}
```

### 6.2 关键实现决策

#### 决策 1: 实时更新机制
**选项 A**: WebSocket /长连接
**选项 B**: Service Worker 推送 + 轮询
**建议**: **选项 B** - Chrome 扩展环境更适合使用 Service Worker 事件配合短轮询

#### 决策 2: 面板收起时的数据保留
**选项 A**: 完全卸载组件，保留状态
**选项 B**: 保持组件挂载，CSS 隐藏
**建议**: **选项 A** - 减少内存占用，重新展开时从状态恢复

#### 决策 3: 多讨论并发限制
**选项 A**: 单讨论队列，顺序执行
**选项 B**: 多讨论并行，每个讨论独立轮询
**建议**: **选项 B** - 用户可同时启动多个独立讨论，各自有进度

### 6.3 性能优化

1. **虚拟滚动**: 讨论消息多时只渲染可视区域
2. **节流更新**: 进度条更新节流至 200ms
3. **懒加载**: 历史讨论列表按需加载
4. **缓存策略**: 已完成讨论的完整内容本地缓存

### 6.4 错误处理

| 场景 | 处理方案 |
|------|----------|
| 模型 API 超时 | 显示重试按钮，保留已获取结果 |
| 网络断开 | 自动重试 3 次，失败显示离线状态 |
| 扩展被重启 | 恢复时检查讨论状态，标记"可能中断" |

## 7. 已确认决策

以下决策已确认：

### 决策 1: 多讨论并发数量限制
- **已确认**: 选项 A - 最多同时运行 3 个讨论
- **理由**: 平衡用户体验与资源占用，避免过多并发导致性能问题

### 决策 2: 已完成讨论的保留策略
- **已确认**: 选项 A - 自动保留最近 20 个
- **理由**: 兼顾历史查阅需求与存储空间管理

### 决策 3: 通知权限策略
- **已确认**: 选项 A - 首次启动时请求
- **理由**: 确保用户及时获得讨论完成通知
